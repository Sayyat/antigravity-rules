# Smart I18n Workflow (`smart-i18next-cli`)

This project uses the custom `smart-i18next-cli` tool (built on top of i18next) for advanced localization workflows.

## core Principles

### 1. 1-to-1 Namespacing
*   **Concept**: Localization files mirror the source code structure exactly.
*   **Benefit**: You always know exactly where a translation file lives.
*   **Example**: 
    *   Source: `src/app/dashboard/page.tsx` 
    *   Locale: `src/i18n/locales/en/app.dashboard.page.json`

### 2. Gettext-Style Keys
*   **Concept**: Use the default English text as the translation key.
*   **Syntax**: `t("My value")` instead of `t("my_key")`.
*   **Result**: The CLI automatically generates `{"My value": "My value"}` in the source JSON.
*   **Benefit**: Code remains readable ("My value" vs "common.labels.submit").

### 3. Static Type Generation
*   **Concept**: The CLI generates static Type Definitions (`.d.ts`) based on the JSON files.
*   **Benefit**: `t()` is fully typed. You get compile-time errors if a key is missing or misspelled.

### 4. Auto Machine Translation
*   **Concept**: Uses Deep Translate (Rapid API).
*   **Trigger**: The CLI detects keys where `key === value` (e.g., `{"Hello": "Hello"}`) in non-default languages.
*   **Action**: It automatically sends these to the API and updates the value (e.g., `{"Hello": "Привет"}`).

## Common Commands (CLI)

```bash
# Display help
pnpm i18n help

# Initialize project
pnpm i18n init [--react | --core]

# Generate types and configs
pnpm i18n generate-configs

# Rename a key globally (Refactoring)
pnpm i18n rename-key "Old Text" "New Text"
```
